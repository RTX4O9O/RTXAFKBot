# AGENTS.md — Guide for AI coding agents

Purpose: help automated agents quickly understand, modify, and verify the FakePlayerPlugin codebase.

1) Quick repo map
- `src/main/java/me/bill/fakePlayerPlugin/` — main code and entry points
- `src/main/resources/` — default `config.yml`, `plugin.yml`, `bot-names.yml`, `bot-messages.yml`
- `src/main/resources/language/en.yml` — all user-facing messages (loaded by `Lang.java`)
- `pom.xml` — build + jar metadata (`target/fpp-*.jar` produced)
- `frontend/`, root `package.json`, and `vercel.json` — separate Node/Vercel status + update API sidecar used by the plugin’s update-check fallback; not part of the Maven plugin jar build.
- `wiki/` — feature docs (Commands.md, Configuration.md, Getting-Started.md, Database.md, Language.md, Permissions.md, Skin-System.md, Swap-System.md, Bot-Behaviour.md, Bot-Messages.md, Bot-Names.md, Fake-Chat.md, FAQ.md, Migration.md, Home.md, Proxy-Support.md, Config-Sync.md, Placeholders.md)
- `src/main/java/me/bill/fakePlayerPlugin/gui/` — in-game settings GUI (`SettingGui`): 3-row chest that lets admins toggle booleans and set numeric/double config values without editing `config.yml` directly. Opened via `/fpp settings`. Implements `Listener` and is registered in `onEnable` via `registerEvents(settingGui, this)`.
- `src/main/java/me/bill/fakePlayerPlugin/util/` — helper subsystems and utilities (e.g. `FppMetrics`, `FppLogger`, `FppPlaceholderExpansion`, `TabListManager`, `BackupManager`, `UpdateChecker`, `ConfigValidator`, `DataMigrator`, `TextUtil`, `ConfigMigrator`, `LuckPermsHelper`, `CompatibilityChecker`, `YamlFileSyncer`, `BotTabTeam`).
- `src/main/java/me/bill/fakePlayerPlugin/listener/` — Bukkit event listeners (`PlayerJoinListener`, `FakePlayerEntityListener`, `FakePlayerKickListener`, `BotCollisionListener`, `PlayerWorldChangeListener`, `ServerListListener`, `BotCommandBlocker`, `BotSpawnProtectionListener`). Note: `LuckPermsUpdateListener` was removed — LP now detects NMS bots as real players automatically. `BotCommandBlocker` prevents bots from executing any commands. `BotSpawnProtectionListener` prevents lobby/spawn plugins from teleporting bots during their 5-tick spawn grace period.
- `src/main/java/me/bill/fakePlayerPlugin/fakeplayer/` — bot domain objects and subsystems (e.g. `FakePlayer`, `FakePlayerBody`, `FakePlayerManager`, `BotPersistence`, `ChunkLoader`, `SkinRepository`, `SkinFetcher`, `SkinProfile`, `SkinProfileInjector`, `NmsHelper`, `NmsPlayerSpawner`, `NmsReflection`, `BotSwapAI`, `BotChatAI`, `BotBroadcast`, `PacketHelper`, `RemoteBotCache`, `RemoteBotEntry`, `BotType`, `BotCombatMode`, `BotPvpAI`, `PeakHoursManager`). Sub-package `fakeplayer/network/` contains the fake Netty layer: `FakeChannel` (discard channel used by `FakeConnection`), `FakeChannelPipeline` (no-op pipeline), `FakeConnection` (NMS `Connection` subclass — overrides all three `send()` variants as no-ops to prevent vanilla connection side-effects during `placeNewPlayer()` that previously caused a phantom "Anonymous User" / UUID-0 ghost entry; `isConnected()` always returns `true`; injects `FakeChannel` and calls `Connection.configureSerialization()` on the no-op pipeline to satisfy NMS internal checks), and `FakeServerGamePacketListenerImpl` (no-op SGPL subclass that discards all outbound packets except knockback). Note: `BotHeadAI` has been removed; head-AI functionality (player tracking, look-range, turn-speed) is now driven by config keys `head-ai.*` and executed inside `FakePlayerManager`.
- `src/main/java/me/bill/fakePlayerPlugin/messaging/` — proxy communication (`VelocityChannel`): registers `fpp:main` plugin-messaging channel; wraps outbound messages in a BungeeCord `Forward ALL` envelope so Velocity/BungeeCord re-delivers to every backend. Subchannels: `BOT_SPAWN`, `BOT_DESPAWN`, `BOT_UPDATE` (display-name sync — wire: `[msgId][serverId][uuid][newDisplayName]`; sent by `broadcastBotDisplayNameUpdate(fp)` when LP refreshes a bot's display name), `CHAT`, `ALERT`, `JOIN`, `LEAVE`, `SYNC`. Echo suppression via per-message unique IDs.
- `src/main/java/me/bill/fakePlayerPlugin/sync/` — proxy config sync (`ConfigSyncManager`): push/pull config files to/from MySQL in NETWORK mode. Supports `DISABLED`, `MANUAL`, `AUTO_PULL`, `AUTO_PUSH` modes. Server-specific keys (`database.server-id`, `database.mysql.*`, `debug`) are never synced.
- `src/main/java/me/bill/fakePlayerPlugin/database/` — session history and analytics (`DatabaseManager`, `BotRecord`).
- `src/main/java/me/bill/fakePlayerPlugin/lang/` — message rendering (`Lang`).
- `src/main/java/me/bill/fakePlayerPlugin/permission/` — centralised permission registry (`Perm`).

2) Where to start (entry points)
- `FakePlayerPlugin.java` — plugin lifecycle (`onEnable` / `onDisable`) and orchestration.
- `command/CommandManager.java` and command classes (`SpawnCommand`, `DeleteCommand` (registers as `despawn`), `ListCommand`, `HelpCommand`, `InfoCommand`, `ChatCommand`, `SwapCommand`, `ReloadCommand`, `FreezeCommand`, `StatsCommand`, `MigrateCommand`, `TpCommand`, `TphCommand`, `LpInfoCommand`, `RankCommand`, `AlertCommand`, `SyncCommand`, `PeaksCommand`, `SettingCommand`) — command wiring and tab completion.
- `command/FppCommand.java` — interface every sub-command implements (`getName`, `getUsage`, `getDescription`, `getPermission`, `canUse`, `execute`, `tabComplete`).
- `config/Config.java`, `util/ConfigMigrator.java` — configuration pattern and automatic migration.
- `fakeplayer/FakePlayerManager.java`, `fakeplayer/BotPersistence.java` — bot lifecycle, persistence and restore.
- `fakeplayer/PeakHoursManager.java` — time-window-based bot pool scheduler. Runs a 60-second repeating tick; computes a target fraction of the total bot pool that should be online based on the current time window (`peak-hours.schedule` / `peak-hours.day-overrides`), then gradually sleeps (quietly removes) or wakes (respawns) AFK bots to match that fraction. Stagger spread is configured via `peak-hours.stagger-seconds`. Requires `swap.enabled: true`. Integrates with `BotSwapAI` — swapped-out bots count toward the pool so short swap absences don't distort the math. Sleeping-bot state is persisted crash-safely via `fpp_sleeping_bots` DB table (populated on every queue mutation; cleared on `wakeAll()`). Crash recovery: `restoreSleepingBotsFromDatabase(db)` repopulates the in-memory deque at startup. Hot-reload: `reload()` wakes all sleeping bots then restarts the tick if still enabled. Key public API: `start()`, `shutdown()`, `reload()`, `wakeAll()`, `wakeBotByName(name)`, `putBotToSleepByName(name)`, `forceCheck()`, `getSleepingCount()`, `getTotalPool()`, `getSleepingEntries()`, `getSleepingNames()`, `isSleeping(name)`, `getCurrentWindowLabel()`, `computeTargetFraction()`, `getSecondsToNextWindow()`, `getNextWindowFraction()`, `isRunning()`. Accessed via `plugin.getPeakHoursManager()`. Inner record: `PeakHoursManager.SleepingBot(name, loc)`.
- `fakeplayer/FakePlayer.java` — domain object for a single active bot (name, UUID, profile, spawnLocation, displayName, skinName, frozen state). `getPhysicsEntity()` and `getPlayer()` both return the single NMS ServerPlayer entity. Additional fields: `luckpermsGroup` (per-bot LP group override, `null` = not yet resolved; get/set via `getLuckpermsGroup()`/`setLuckpermsGroup()`), `rawDisplayName` (the `{bot_name}` slot before LP prefix/suffix are added; used as the stable base for `refreshLpDisplayName` rebuilds), `bodyless` (when `true` the bot exists in tab-list/chat only, no entity in world — used for console spawns without location data), `botType` (`BotType` enum — `AFK` = default passive bot; `PVP` = PvP bot (name prefixed `pvp_<name>`, currently restricted to internal dev UUID); get/set via `getBotType()`/`setBotType()`), `chatEnabled` (per-bot chat mute flag, default `true`; `false` = permanently silenced; get/set via `isChatEnabled()`/`setChatEnabled()`), `chatTier` (per-bot chat activity tier override, `null` = random; values `"quiet"`, `"passive"`, `"normal"`, `"active"`, `"chatty"`; get/set via `getChatTier()`/`setChatTier()`). Note: `botIndex` has been removed; `nametagEntity` remains as a legacy-compat field (no-op for NMS players).
- `fakeplayer/FakePlayerBody.java` — NMS ServerPlayer spawning/removal. Entities tagged via PDC keys for identification.
- `fakeplayer/NmsPlayerSpawner.java` — reflection-based NMS `ServerPlayer` factory (cached class/method/constructor lookups); bypasses Mojang auth by creating the player directly in JVM memory.
- `fakeplayer/NmsReflection.java` — shared NMS reflection bootstrap (CraftBukkit/NMS class resolution); initialised once, thread-safe.
- `fakeplayer/RemoteBotCache.java` — thread-safe registry of bots physically running on other proxy servers. Always initialised (even in LOCAL mode). Populated from DB at startup (NETWORK mode) and updated via `BOT_SPAWN`/`BOT_DESPAWN` plugin messages. Exposed via `FakePlayerPlugin.getRemoteBotCache()`.
- `fakeplayer/RemoteBotEntry.java` — immutable record (`serverId`, `uuid`, `name`, `displayName`, `packetProfileName`, `skinValue`, `skinSignature`) for a remote bot; `hasSkin()` checks whether texture data is present.
- `fakeplayer/SkinProfile.java` — immutable snapshot of a resolved Mojang skin (base64 value + signature + source label).
- `config/BotNameConfig.java`, `config/BotMessageConfig.java` — bot name & message pools which are initialised in `onEnable`.
- `fakeplayer/SkinRepository.java` / `fakeplayer/SkinFetcher.java` — skin-loading subsystem (lazy singleton; explicitly initialised via `SkinRepository.get().init(this)` in `onEnable` after BotMessageConfig but before database init; must be ready before any bot spawn including persistence restore).
- `fakeplayer/network/FakeConnection.java` — NMS `Connection` subclass used for every bot. Created before `placeNewPlayer()` via `new FakeConnection(loopbackAddress)`; sets `this.channel = new FakeChannel(...)` and calls `Connection.configureSerialization()` on the no-op pipeline. Overrides all three `send()` overloads as no-ops to eliminate vanilla connection side-effects (protocol-state registers, pending-ack queues, callback notifications) that caused a phantom "Anonymous User" entry (UUID `0000…0000`) in the player list. `isConnected()` always returns `true` so NMS never treats the bot as disconnected.
- `fakeplayer/network/FakeServerGamePacketListenerImpl.java` — extends NMS `ServerGamePacketListenerImpl`; `send()` is a no-op that discards all outbound packets. Exceptions: `ClientboundSetEntityMotionPacket` (knockback) is intercepted and applied server-side via reflected `getMovement()`/`lerpMotion()` (1.21.9+) or `getXa()/Ya()/Za()` fallback. Injected into `ServerPlayer.connection` immediately after `placeNewPlayer()` so `awaitingPositionFromClient` stays `null` — preventing the server from snapping the bot back to its spawn position.
- `fakeplayer/SkinProfileInjector.java` — reflection helper that injects a `textures` property into an authlib `GameProfile`; used by both NMS body spawns and tab-list packet profiles for consistent skin application. Call `SkinProfileInjector.apply(gameProfile, skinProfile)`.
- `fakeplayer/BotIdentityCache.java` — persistent registry mapping `(bot_name, server_id) → stable UUID`. Ensures a bot with the same name always rejoins with the same UUID (LuckPerms data, inventory, session history stay linked). Storage priority: (1) in-memory `ConcurrentHashMap` (always checked first), (2) `fpp_bot_identities` DB table when DB is enabled, (3) `data/bot-identities.yml` YAML fallback when DB is disabled. Initialized in `onEnable` after the DB, set on `FakePlayerManager` via `setIdentityCache()`; `FakePlayerManager.resolveUuid(name)` delegates here for every fresh spawn. `prime(name, uuid)` pre-populates the cache from `spawnRestored()` so restart-restored bots don't hit the DB a second time. **Identity scope:** per-server — cross-server UUID sharing is intentionally not supported. Keys are lowercased for case-insensitive lookups.
- `gui/SettingGui.java` — interactive 3-row chest GUI (`/fpp settings`). 5 categories: General, Body, Chat, Swap, Peak Hours; up to 18 settings per page. Toggle items flip their boolean in-place on click; numeric/double items close the chest and prompt the player to type a value in chat (message cancelled — invisible to other players, 60-second timeout auto-cancels). After each change, calls `plugin.saveConfig()`, `Config.reload()`, and `applyLiveEffect(configKey)` — which mirrors `ReloadCommand`'s subsystem calls for instant live effect without a full `/fpp reload`. **Reset button** (slot 18): resets all settings on the current page to JAR defaults. Per-player session state stored in `sessions` map (keyed by UUID). Chat-input sessions tracked separately in `chatSessions`; `pendingChatInput` / `pendingRebuild` sets prevent false `InventoryCloseEvent` handling during transitions. `SettingGui` is a singleton instantiated in `onEnable`, passed to `SettingCommand`, and registered as a listener via `registerEvents(settingGui, this)`. **Permission:** `Perm.SETTINGS = "fpp.settings"` (default `op`).
- `lang/Lang.java` — message rendering. Loads `language/en.yml`, supports named and positional placeholders. Use `Lang.get("key", "name", "value")` for an Adventure `Component` or `Lang.raw("key", ...)` for a raw string. Hot-reloadable via `/fpp reload`.
- `permission/Perm.java` — central permission registry. All permission nodes declared as constants (e.g. `Perm.SPAWN`, `Perm.DELETE`, `Perm.USER_SPAWN`). Use `Perm.has()`, `Perm.hasOrOp()`, `Perm.resolveUserBotLimit()` helpers. Never hard-code permission strings in commands.
- `database/DatabaseManager.java`, `database/BotRecord.java` — session history and analytics. Supports SQLite (default, WAL mode) and MySQL. Writes are serialised on a background thread; positions are batch-flushed on a configurable interval. Controlled by `Config.databaseEnabled()` and `Config.mysqlEnabled()`. **Schema version 8.** Tables: `fpp_bot_sessions` (session history), `fpp_active_bots` (restart persistence source of truth), `fpp_sleeping_bots` (peak-hours crash-safe sleeping-bot queue — `sleep_order`, `bot_name`, `world_name`, `pos_x/y/z`, `pos_yaw/pitch`, `server_id`; populated whenever the peak-hours sleep queue changes, cleared on `wakeAll()`), `fpp_bot_identities` (stable name→UUID registry — `bot_name`, `server_id`, `bot_uuid`, `created_at`; PRIMARY KEY `(bot_name, server_id)`; backfilled from earliest session history on schema upgrade), `fpp_meta` (schema version). API: `saveSleepingBots(List<SleepingBotRow>)`, `loadSleepingBots()`, `clearSleepingBots()`, `lookupBotUuid(botName, serverId)` (synchronous read → `UUID` or `null`), `registerBotUuid(botName, uuid, serverId)` (enqueued write — INSERT OR IGNORE so first UUID always wins).
- `util/TabListManager.java`, `util/UpdateChecker.java` — tab header/footer management and async update checker initialized during startup.

3) Key patterns & concrete examples (copy-ready pointers)
- Migration runs before config init: `ConfigMigrator.migrateIfNeeded(this)` in `onEnable` (see `FakePlayerPlugin.java`).
- Config usage: call `Config.init(this)` then use accessors like `Config.fakeChatEnabled()` in code. Tab-list visibility accessor: `Config.tabListEnabled()` (key `tab-list.enabled`, default `true`) — when `false`, bots are never added to any player's tab list; the 20-tick display-name refresh loop, `syncToPlayer`, and the respawn re-add in `FakePlayerEntityListener` all check this flag before sending any `sendTabListAdd` packets. Toggling via `/fpp reload` calls `FakePlayerManager.applyTabListConfig()` which immediately removes or re-adds all active bots' tab entries for all online players. **Key accessors:** `Config.fakeChatEnabled()` (`fake-chat.enabled`, default `false`); `Config.fakeChatRequirePlayer()` (`fake-chat.require-player-online`, default `true`); `Config.fakeChatMessages()` (messages from `bot-messages.yml`). Local bots with a body use `player.chat(message)` so the server's real chat pipeline handles formatting. Bodyless bots and remote proxy bots use `Config.fakeChatRemoteFormat()` (`fake-chat.remote-format`, default `"<yellow>{name}<dark_gray>: <white>{message}"`) — supports `{name}` and `{message}` placeholders. `Config.bodyPushable()` (`body.pushable`, default `true`); `Config.bodyDamageable()` (`body.damageable`, default `true`); `Config.warningsNotifyAdmins()` (`messages.notify-admins-on-join`, default `true`) — send compatibility warning to online admins on startup. **Granular debug flags:** `Config.debugStartup()`, `Config.debugNms()`, `Config.debugPackets()`, `Config.debugLuckPerms()`, `Config.debugNetwork()`, `Config.debugConfigSync()`, `Config.debugSkin()`, `Config.debugDatabase()`, `Config.debugChat()` — each maps to `logging.debug.<category>` in config (also triggered by global `debug: true`). **Network/proxy accessors:** `Config.databaseMode()` (`database.mode`, `"LOCAL"` or `"NETWORK"`); `Config.isNetworkMode()` — shorthand for `databaseEnabled() && databaseMode()=="NETWORK"`; `Config.serverId()` (`database.server-id`, falls back to legacy `server.id`); `Config.configSyncMode()` (`config-sync.mode`, one of `DISABLED` / `MANUAL` / `AUTO_PULL` / `AUTO_PUSH`). Server-specific keys (`database.server-id`, `database.mysql.*`, `debug`) are never pushed/pulled by `ConfigSyncManager`. Note: `Config.tabListNameFormat()` and the `bot-name.tab-list-format` config key were **removed** in config v38 — LP manages prefix/suffix natively for NMS ServerPlayer entities and the display name packet carries only the bot name itself.
- Language messages: always use `Lang.get("key")` (returns Adventure `Component`) or `Lang.raw("key")` (returns `String`) — never hard-code user-facing text. Named placeholders: `Lang.get("freeze-frozen", "name", fp.getDisplayName())`. All keys live in `src/main/resources/language/en.yml`.
- Permission checks: use `Perm.has(sender, Perm.SPAWN)` or `Perm.hasOrOp(sender, Perm.ALL)` — never hard-code node strings. **All permissions declared in `Perm.java` as constants AND in `plugin.yml` for LuckPerms auto-discovery.** Per-user bot limits resolved via `Perm.resolveUserBotLimit(sender)` (scans `fpp.bot.1` … `fpp.bot.100`; returns -1 if none set, caller falls back to `Config.userBotLimit()`). When adding new commands: (1) add constant to `Perm.java`, (2) add full entry to `plugin.yml` with description and default, (3) add to `fpp.*` children list in `plugin.yml`, (4) use constant in command's `getPermission()` and `canUse()` methods. **LuckPerms tab-completion depends on `plugin.yml` declarations** — missing entries won't appear in `/lp permission set fpp.<tab>`.
- Command registration: plugin registers commands via `commandManager.register(new SpawnCommand(...))`; `HelpCommand` is auto-registered inside `CommandManager`, and `/fpp help` reads live from the registered command list, so new sub-commands need no separate help-menu wiring. The plugin then calls `getCommand("fpp").setExecutor(commandManager)` (see `onEnable`).
- Listener registration: `getServer().getPluginManager().registerEvents(new PlayerJoinListener(this, fakePlayerManager), this)` — listeners receive plugin + manager references.
- Persistence lifecycle: restore on enable via `botPersistence.purgeOrphanedBodiesAndRestore(fakePlayerManager)` (purges orphaned entities then calls `restore()` 5 ticks later) and save on disable via `botPersistence.save(fakePlayerManager.getActivePlayers())`. `BotPersistence.saveAsync()` also available for non-shutdown saves (snapshots locations on main thread, writes async). When DB is enabled, the `fpp_active_bots` table is the **primary** restore source (`DatabaseManager.getActiveBots()`); `data/active-bots.yml` is the fallback for no-DB or empty-table cases.
- Metrics & soft-deps: metrics are initialized with `FppMetrics` when `Config.metricsEnabled()`; PlaceholderAPI expansion registered conditionally in `onEnable`.
- Compatibility checks: `CompatibilityChecker.check()` (in `util/CompatibilityChecker.java`) **always returns a non-restricted result** — the old Paper-detection and minimum MC version gate has been removed. `check()` is **not called in `onEnable`**; `compatibilityRestricted` and `compatibilityWarningMessage` fields have been removed from `FakePlayerPlugin`. The class retains two general-purpose utilities: `CompatibilityChecker.extractMcVersion()` (returns detected MC version string) and `CompatibilityChecker.isVersionAtLeast(version, required)` (component-wise numeric comparison). `buildWarningComponent(result)` always returns `null`.
- Bot pools: `BotNameConfig.init(this)` and `BotMessageConfig.init(this)` are called during startup — the plugin loads `bot-names.yml` and `bot-messages.yml` early.
- Metrics / FastStats packaging: `FppMetrics` is initialized before the startup banner to reflect metrics status. FastStats jars are bundled under `src/main/resources/faststats/` (the POM treats them as binary resources, not shaded). Note: building requires JDK 21 (see `pom.xml` <java.version>21).
- Dependencies: **PacketEvents 2.x is required** — FPP uses it for tab-list packet integration. Declared in `plugin.yml` as a soft-depend: `PlaceholderAPI` and `LuckPerms` (both optional, auto-detected).
- Entity stack: each bot is a single NMS ServerPlayer entity (created via `NmsPlayerSpawner`) with full player functionality including chat, tab-list presence, and realistic movement. Use PDC keys for identification during event handling.
- Database gate: check `Config.databaseEnabled()` before any `DatabaseManager` call; `databaseManager` may be `null` at runtime (disabled or failed to init). `databaseManager.recordAllShutdown()` must be called in `onDisable` before `databaseManager.close()`.
- Text formatting: `TextUtil.colorize(string)` parses MiniMessage tags and legacy `&`/`§` codes into an Adventure `Component`. `TextUtil.format(string)` additionally applies small-caps Unicode conversion. **Supports all modern color formats:** MiniMessage tags (`<rainbow>`, `<gradient:#FF0000:#0000FF>`), hex colors (`<#9782ff>`), LuckPerms gradient shorthand (`{#fffff>}text{#00000<}`), and mixed formats (`&7[<#9782ff>Phantom</#9782ff>&7]`).
- LuckPerms integration: `LuckPermsHelper` reads resolved prefix/suffix directly from LP's in-memory `CachedMetaData` — bots are real NMS `ServerPlayer` entities, LP handles them natively. Core helpers: `LuckPermsHelper.isAvailable()`, `LuckPermsHelper.groupExists(groupName)`, `LuckPermsHelper.getAllGroupNames()`, `LuckPermsHelper.buildGroupSummary()`. Runtime group assignment: `LuckPermsHelper.setPlayerGroup(uuid, groupName)` (works online or offline) or `LuckPermsHelper.applyGroupToOnlineUser(uuid, groupName)` (prefers the live `User` instance so `UserDataRecalculateEvent` fires immediately — preferred when bot is already spawned). Prefix/suffix reads: `LuckPermsHelper.getResolvedPrefix(uuid)` / `LuckPermsHelper.getResolvedSuffix(uuid)` (synchronous, requires bot to be online). Async group read from storage: `LuckPermsHelper.getStoredPrimaryGroup(uuid)`. Online-only read: `LuckPermsHelper.getPrimaryGroup(uuid)`. Force LP metadata refresh: `LuckPermsHelper.refreshUserCache(uuid)`. `LuckPermsHelper.invalidateCache()` is `@Deprecated` (no-op — LP handles caching natively). **Event subscription:** `LuckPermsHelper.subscribeLpEvents(plugin, manager)` subscribes to `UserDataRecalculateEvent` so bot display names auto-update whenever a group changes (LP command, API, etc.); called directly in `onEnable` (not deferred). `LuckPermsHelper.unsubscribeLpEvents()` is called in `onDisable` to prevent memory leaks. **LP group assignment:** `Config.luckpermsDefaultGroup()` (`luckperms.default-group`, default `""`) — optional LP group assigned at spawn; blank = `"default"`. Use `/fpp lpinfo [bot-name]` for in-game diagnostics. Check LP availability via `LuckPermsHelper.isAvailable()` before any API call. **LP injection flow:** `finishSpawn` calls `LuckPermsHelper.ensureGroupBeforeSpawn(botUuid, cfgGroup)` **async before the NMS body spawns** — writes the group to LP storage so `PlayerJoinEvent` listeners (TAB plugin, etc.) see the correct group immediately. If the bot already has a non-default group in storage (prior `/fpp rank`), that group is preserved unless `luckperms.default-group` overrides it. Once LP completes, `spawnBodyAndFinish(fp, spawnLoc)` runs on the main thread. After the body spawns, `LuckPermsHelper.applyGroupToOnlineUser(uuid, group)` is called 5 ticks later; after that completes, `refreshLpDisplayName(fp)` is called 2 ticks later to update the tab-list display name. `FakePlayer.rawDisplayName` is the stable `{bot_name}` base for rebuilds. Call `FakePlayerManager.refreshLpDisplayName(fp)` to manually rebuild display name + resend tab-list update packets. **Scoreboard team override:** All bots are placed in the `~fpp` scoreboard team (managed by `BotTabTeam`) which sorts AFTER all other teams, ensuring bots always appear below real players regardless of LP group weight. Note: `LpData` record, `getBotLpData()`, `buildPacketProfileName()`, `invalidateGroup()`, `refreshRankList()`, and the packet-prefix-char weight-rank ordering system have all been removed. **ClassLoader guard:** `FakePlayerPlugin` caches a `luckPermsAvailable` boolean (set once in `onEnable()` via `Bukkit.getPluginManager().getPlugin("LuckPerms") != null`) accessible via `plugin.isLuckPermsAvailable()`. **Never reference `LuckPermsHelper` or any LP API class unless this returns `true`** — Paper's `PluginClassLoader` eagerly resolves LP API class references during bytecode verification, so even calling `LuckPermsHelper.isAvailable()` on a server without LP installed throws `NoClassDefFoundError`. The pattern everywhere LP is needed: `if (plugin.isLuckPermsAvailable()) { LuckPermsHelper.doSomething(...); }`.
- Listener registration: `getServer().getPluginManager().registerEvents(new PlayerJoinListener(this, fakePlayerManager), this)` — listeners receive plugin + manager references. `FakePlayerKickListener` (guards all bots from automated kicks like anti-idle/anti-AFK) is always registered; all other listeners (`FakePlayerEntityListener`, `BotCollisionListener`, etc.) are also always registered — the old `compatibilityRestricted` skip is gone.
- `onDisable` sequence: `chunkLoader.releaseAll()` → `botChatAI.cancelAll()` → **`peakHoursManager.shutdown()`** (wakes all sleeping bots so they appear in `activePlayers`) → `botSwapAI.cancelAll()` → **save persistence** (sleeping bots are now active and captured) → `LuckPermsHelper.unsubscribeLpEvents()` → `fakePlayerManager.removeAllSync()` (also calls `pvpAI.cancelAll()` internally) → `tabListManager.shutdown()` → `botTabTeam.destroy()` → `databaseManager.recordAllShutdown()` + `close()` → `fppMetrics.shutdown()` → unregister VelocityChannel messaging channels. **Critical ordering:** `peakHoursManager.shutdown()` MUST run before `botPersistence.save()` so sleeping bots are woken and included in the persistence file.
- Update notifications to late-joining admins: stored as a `Component` on `FakePlayerPlugin.getUpdateNotification()` / `setUpdateNotification()`; `PlayerJoinListener` delivers it to players with `Perm.ALL`.
- YAML file sync: `YamlFileSyncer.syncMissingKeys(plugin, diskRelPath, jarResourcePath)` — non-destructive; adds keys present in the JAR but absent from disk, never overwrites user values. Returns a `SyncResult` record (`fileName`, `keysAdded`, `hasChanges()`, `count()`). Called automatically inside `Lang.reload()`, `BotNameConfig.reload()`, and `BotMessageConfig.reload()` on every startup and `/fpp reload`. Force-run from the command line with `/fpp migrate lang`, `/fpp migrate names`, or `/fpp migrate messages`. When adding new keys to `en.yml`, `bot-names.yml`, or `bot-messages.yml`, they will be automatically synced to existing user files on next reload — no manual migration step needed.
- Spawn cooldown: `Config.spawnCooldown()` (key `spawn-cooldown`, default `0` = disabled). Applies to both admin and user `/fpp spawn`. Players with `Perm.BYPASS_COOLDOWN` (`fpp.bypass.cooldown`) are exempt. Cooldown state lives in `FakePlayerManager.spawnCooldowns` (per-UUID map). Helpers: `manager.isOnCooldown(uuid)`, `manager.getRemainingCooldown(uuid)` (seconds), `manager.recordSpawnCooldown(uuid)`.
- Backup variants: `BackupManager.createFullBackup(plugin, reason)` — backs up all plugin files including SQLite; used by `ConfigMigrator` before migrations. `BackupManager.createConfigFilesBackup(plugin, reason)` — backs up YAML files only (`config.yml`, `bot-names.yml`, `bot-messages.yml`, `language/`); used before lightweight sync operations (e.g. `/fpp migrate lang`). Both methods prune to the most recent `MAX_BACKUPS` (10) automatically.
- Additional `Perm` helpers: `Perm.missing(sender, node)` returns `true` when the sender lacks the node (negation of `has()`); `Perm.hasAny(sender, node1, node2, ...)` returns `true` when the sender has at least one of the listed nodes.
- Physical-body API: `FakePlayerManager.physicalBodiesEnabled()` — returns `Config.spawnBody()`. (The old `&& !plugin.isCompatibilityRestricted()` check has been removed.) Use this instead of checking `Config.spawnBody()` inline when deciding whether NMS ServerPlayer bodies are active.
- Display-name pipeline: `FakePlayerManager.finalizeDisplayName(rawName, botName)` — private helper that expands PAPI `%placeholders%` via `PlaceholderAPI.setPlaceholders(null, ...)` (null player = server-wide), then calls `sanitizeDisplayName()`. **Note:** the `{prefix}/{bot_name}/{suffix}` tab-list format substitution (previously driven by `bot-name.tab-list-format`) was removed in config v38 — LP manages prefix/suffix natively for real NMS ServerPlayer entities; the display name packet carries only the bot name. Called at every display-name build site: `spawnUserBot`, `spawn`, `spawnRestored`, `validateUserBotNames`. Public entry point for post-LP-assignment rebuilds: `FakePlayerManager.refreshLpDisplayName(fp)` (also has a private retry variant `refreshLpDisplayNameWithRetry` for LP metadata race conditions).
- Body damageable/pushable live-reload: `FakePlayerManager.applyBodyConfig()` — called on `/fpp reload`; handles damage/pushable configuration for all active NMS ServerPlayer entities. `BotCollisionListener` also guards all three push paths (`onEntityDamageByEntity`, `onPlayerMove`, `tickBotSeparation`) with `if (!Config.bodyPushable()) return;`.
- Swim AI: `Config.swimAiEnabled()` (`swim-ai.enabled`, default `true`). Each tick the main loop calls `NmsPlayerSpawner.setJumping(bot, bot.isInWater() || bot.isInLava())` so bots automatically swim upward when submerged, mimicking a player holding spacebar. Set `swim-ai.enabled: false` to let bots sink/drown instead.
- PVP AI subsystem: `BotPvpAI` drives all movement and combat for `BotType.PVP` bots. Initialized in `FakePlayerManager` constructor; accessed via `manager.getPvpAI()`. The main tick loop calls `pvpAI.tickBot(fp, bot, online)` each game tick for PVP bots only (head-AI is skipped for PVP bots — `BotPvpAI` handles rotation internally). Key config: `Config.pvpAiDifficulty()` (`pvp-ai.difficulty`, default `"medium"`, values: `easy`/`medium`/`hard`/`tier1`/`hacker`); `Config.pvpAiDefensiveMode()` (`pvp-ai.defensive-mode`, default `true` — bots only attack back when hit); `Config.pvpAiDetectRange()` (`pvp-ai.detect-range`, default `32.0`). When a player damages a PVP bot, `FakePlayerEntityListener` calls `manager.getPvpAI().onBotAttacked(botUuid, attackerUuid)` to register the attacker for defensive retaliation. Combat mode (`BotCombatMode.CRYSTAL`/`SWORD`/`FIST`) is auto-detected from the bot's inventory every 2 seconds. Difficulty controls attack range, crit/s-tap consistency, and early-swing window. **⚠ PVP bots are currently restricted to the internal developer UUID — not yet available for general spawning.**
- BotChatAI expanded features: Each bot is assigned a random activity tier at first use — quiet (15%), passive (25%), normal (30%), active (18%), chatty (12%) — which scales the chat interval multiplier. Override per-bot via `FakePlayer.setChatTier(String)` or via `/fpp chat <bot> tier <tier>`; reset to random with `tier reset`. Silence a bot permanently (`/fpp chat <bot> off` or `fp.setChatEnabled(false)`) or temporarily (`/fpp chat <bot> mute [seconds]` — `BotChatAI.timedMute(uuid, seconds)`). Inspect state with `/fpp chat <bot> info` (shows enabled, tier, multiplier). Bulk operations: `/fpp chat all <on|off|tier <tier>|mute [seconds]>`. **Chat realism config (v39→v40):** `Config.fakeChatTypingDelay()` (`fake-chat.typing-delay`, default `true`), `Config.fakeChatBurstChance()` (`fake-chat.burst-chance`, default `0.12`), `Config.fakeChatBurstDelayMin()/Max()` (`fake-chat.burst-delay.min/max`, default `2/5`), `Config.fakeChatReplyToMentions()` (`fake-chat.reply-to-mentions`, default `true`), `Config.fakeChatMentionReplyChance()` (`fake-chat.mention-reply-chance`, default `0.65`), `Config.fakeChatReplyDelayMin()/Max()` (`fake-chat.reply-delay.min/max`, default `2/8`), `Config.fakeChatStaggerInterval()` (`fake-chat.stagger-interval`, default `3`), `Config.fakeChatActivityVariation()` (`fake-chat.activity-variation`, default `true`), `Config.fakeChatHistorySize()` (`fake-chat.history-size`, default `5`). Event-triggered chat config: `fake-chat.event-triggers.enabled`, `.on-player-join.*`, `.on-death.*`, `.on-player-leave.*`. Keyword reactions: `fake-chat.keyword-reactions`. **`BotMessageConfig` message pools** (all in `bot-messages.yml`, auto-synced via `YamlFileSyncer`): `messages` (base chat pool), `replies` (mention-reply pool — `Config.chatReplyMessages()`), `burst-followups` (burst follow-up pool — `Config.chatBurstMessages()`), `join-reactions` / `death-reactions` / `leave-reactions` / `keyword-reactions.<key>` (event-triggered pools — `Config.chatJoinReactionMessages()`, `chatDeathReactionMessages()`, `chatLeaveReactionMessages()`, `chatKeywordReactionMessages(key)`). Accessed via `FakePlayerPlugin.getBotChatAI()` or `FakePlayerManager.getBotChatAI()`.
- Peak-hours subsystem: `PeakHoursManager` controls time-window-based bot pool scheduling. Requires `swap.enabled: true`. **Config accessors:** `Config.peakHoursEnabled()` (`peak-hours.enabled`, default `false`); `Config.peakHoursTimezone()` (`peak-hours.timezone`, default `"UTC"`); `Config.peakHoursStaggerSeconds()` (`peak-hours.stagger-seconds`, default `30`); `Config.peakHoursSchedule()` (`peak-hours.schedule` — list of `{start, end, fraction}` maps); `Config.peakHoursDayOverrides()` (`peak-hours.day-overrides` — per-`DayOfWeek` schedule overrides, keyed by uppercase `DayOfWeek` name); `Config.peakHoursMinOnline()` (`peak-hours.min-online`, default `0` = no floor); `Config.peakHoursNotifyTransitions()` (`peak-hours.notify-transitions`, default `false` — broadcasts window changes to players with `Perm.PEAKS`). **Permission:** `Perm.PEAKS = "fpp.peaks"`. **Command:** `/fpp peaks [on|off|status|next|force|list|wake [name]|sleep <name>]` — toggle, inspect, or manually override the sleeping pool. **Hot-reload:** `PeakHoursManager.reload()` is called by `ReloadCommand` (step 3b) — wakes all sleeping bots, resets transition state, then restarts the tick if still enabled. **`onDisable` ordering:** `peakHoursManager.shutdown()` wakes all sleeping bots BEFORE `botPersistence.save()` so the full pool is captured. **Crash recovery:** `peakHoursManager.restoreSleepingBotsFromDatabase(db)` is called after `botPersistence.purgeOrphanedBodiesAndRestore()` to repopulate the sleeping-bot deque from `fpp_sleeping_bots`; only runs when `databaseManager != null && Config.persistOnRestart()`.
- `FakePlayer.getLiveLocation()` — returns the most accurate current location: the live NMS Player body position when valid, otherwise the last recorded `spawnLocation`. Used by the DB flush loop in `FakePlayerManager` and anywhere a fresh location is needed without relying on stored state.
- `FakePlayerManager.restorationInProgress` — boolean flag (with `isRestorationInProgress()` / `setRestorationInProgress()`) that is `true` during bot persistence restore and cleared after LP prefix refresh completes. Check this before triggering actions that should not run while bots are being restored.

4) Common tasks & minimal steps (how to implement typical changes)
- Add a new command:
  - Create `src/main/java/me/bill/fakePlayerPlugin/command/MyCommand.java` implementing the command interface used by `CommandManager`.
  - Register it in `FakePlayerPlugin.onEnable()` with `commandManager.register(new MyCommand(...))`.
  - No manual help update is needed — `HelpCommand` reads live from `CommandManager`.
  - Ensure `plugin.yml` contains the `fpp` command or sub-command docs (command parsing done by `CommandManager`).
- Add a config option:
  - Add default to `src/main/resources/config.yml`.
  - If needed, update `Config` accessor and `ConfigMigrator` to preserve old configs.
  - New options should be validated in `util/ConfigValidator.java` if they can be misconfigured.
- Add a language message:
  - Add the key and default text to `src/main/resources/language/en.yml`.
  - Access it in code with `Lang.get("your-key")` or `Lang.raw("your-key", "placeholder", "value")`.
  - The file is hot-reloadable — `/fpp reload` picks up changes without restart.
- Add a permission node:
  - Declare a new `public static final String` constant in `permission/Perm.java` (e.g. `public static final String MY_FEATURE = "fpp.myfeature";`).
  - Add a full entry under `permissions:` in `plugin.yml` with description and default value (e.g. `op`, `true`, or `false`).
  - Add the node to the `fpp.*` wildcard's `children:` list in `plugin.yml` (so admins inherit it automatically).
  - Use the constant in your command's `getPermission()` and `canUse()` methods: `Perm.has(sender, Perm.MY_FEATURE)`.
  - **Critical:** LuckPerms reads `plugin.yml` at startup for tab-completion — permissions missing from `plugin.yml` won't appear in `/lp permission set fpp.<tab>`.
  - Test: restart server, run `/lp user <name> permission set fpp.` and verify your new permission appears in tab-completion.

    - If changing skin behaviour or adding skins: ensure `SkinRepository`/`SkinFetcher` are respected — add skin files to `src/main/resources/skins/` for defaults or `plugins/FakePlayerPlugin/skins/` at runtime and call `/fpp reload` to refresh. The startup code also drops a `skins/README.txt` inside the plugin data folder on first run. **Note (v1.5.4):** `skin.guaranteed-skin` default changed from `true` → `false`; bots with non-Mojang names now use Steve/Alex skins by default. `skin.fallback-pool` (`Config.skinFallbackPool()`) and `skin.fallback-name` (`Config.skinFallbackName()`) still exist as auto-mode fallbacks for guaranteed-skin resolution.

5) Build / run / verify (Windows PowerShell)
Build, obfuscate, AND auto-deploy to `%USERPROFILE%\Desktop\dmc\plugins\fpp.jar` in one command:
```powershell
mvn -DskipTests clean package
```
The `maven-antrun-plugin` in `pom.xml` automatically copies `target/fpp-*-obfuscated.jar` to the deploy folder after ProGuard finishes. To deploy to a different path override the property:
```powershell
mvn -DskipTests clean package "-Ddeploy.dir=C:\path\to\server\plugins"
```
Note: requires JDK 21+ to compile (`pom.xml` `<java.version>21`). FastStats jars are bundled as binary resources in `src/main/resources/faststats/` (not shaded).

**IntelliJ build tool ("Build Artifacts"):** IntelliJ's native artifact builder bypasses Maven and ProGuard. Treat `target/fpp-*-obfuscated.jar` as the source artifact and use `pom.xml` as the source of truth for the current version; some repo docs/scripts still mention `1.4.22`. Workflow:
1. Run the **"Build Plugin (Obfuscated)"** Maven run configuration (or `mvn clean package`) — this runs ProGuard and auto-deploys
2. Optionally run **Build › Build Artifacts** to re-pack the already-obfuscated JAR to the output path without recompiling
- ⚠️ "Build Artifacts" alone (without Maven running first) will fail if `target/fpp-*-obfuscated.jar` does not exist

Double-click `BUILD.bat` in the project root as an alternative to Maven in terminal.

For the optional Node/Vercel sidecar (`frontend/`), use the root `package.json` scripts: `npm install`, then `npm start` (local Express server via `frontend/index.js`) or `npm run dev`.

6) Debugging tips
- To reproduce startup ordering bugs, inspect `onEnable()` in `FakePlayerPlugin.java` — migration → config → Lang → BotNameConfig/BotMessageConfig → ensureDataDirectories → SkinRepository.init → database → **BotIdentityCache** → remote bot cache (NETWORK mode: pre-populate from DB) → ConfigSyncManager (NETWORK mode only) → FakePlayerManager+subsystems → commands (**SettingGui instantiated here; `SettingCommand` registered; `settingGui` registered as listener immediately after**) → listeners → **BotChatAI + BotSwapAI + PeakHoursManager init** (PeakHoursManager started if `peak-hours.enabled && swap.enabled`) → VelocityChannel (plugin messaging) → periodic health check (6000 ticks) → ConfigValidator → TabListManager → BotTabTeam → PlaceholderAPI → LuckPerms (`subscribeLpEvents` — direct, not deferred) → UpdateChecker (purely async, never blocks startup) → Metrics → startup banner → persistence restore → **PeakHoursManager crash recovery** (`restoreSleepingBotsFromDatabase` — only when DB enabled and `persist-on-restart: true`). Note: `CompatibilityChecker.check()` is no longer called at startup — all features are enabled unconditionally.
- To debug config migrations, read `util/ConfigMigrator.java` and check backups under `plugins/FakePlayerPlugin/backups/` created by `BackupManager`.
- For packet-level issues (spawn / entity problems) inspect `fakeplayer/PacketHelper.java` and `fakeplayer/FakePlayerManager.java`.
- For database issues: SQLite file is at `plugins/FakePlayerPlugin/data/fpp.db`; run `/fpp migrate status` to check schema version, YAML sync health, and backup count. Full `/fpp migrate` subcommands: `backup` (create manual backup), `backups` (list stored backups), `status` (config version + file sync health + DB stats), `config` (re-run config migration chain), `lang` / `names` / `messages` (force-sync the respective YAML file from JAR), `db merge [file]` / `db export` / `db tomysql`. Set `database.enabled: false` in config to disable all DB I/O.
- Tab list header/footer and update checker: `TabListManager.start()` and `UpdateChecker.check(this)` are run during `onEnable()` — useful when diagnosing missing tab headers or unexpected update-checker output.

7) Integration & soft-dependencies
- PlaceholderAPI: `util.FppPlaceholderExpansion` registered if plugin present. **Full placeholder list (29+ total)** — server-wide: `%fpp_count%` (bot count — local + remote in NETWORK mode), `%fpp_local_count%` (this server's bots only), `%fpp_network_count%` (remote servers' bots only), `%fpp_max%` (global cap, `∞` if 0), `%fpp_real%` (non-bot players online), `%fpp_total%` (real + bots, network-wide), `%fpp_online%` (alias for `%fpp_total%`), `%fpp_frozen%`, `%fpp_names%` (comma-joined display names — local + remote in NETWORK mode), `%fpp_network_names%` (remote-server bot names only), `%fpp_chat%`, `%fpp_swap%`, `%fpp_skin%`, `%fpp_body%`, `%fpp_pushable%`, `%fpp_damageable%`, `%fpp_tab%`, `%fpp_max_health%`, `%fpp_persistence%`, `%fpp_version%`. Network/Proxy: `%fpp_network%` (on/off for NETWORK mode), `%fpp_server_id%`, `%fpp_spawn_cooldown%`. Per-world: `%fpp_count_<world>%`, `%fpp_real_<world>%`, `%fpp_total_<world>%`. Player-relative: `%fpp_user_count%` (bots owned by that player), `%fpp_user_max%` (personal limit), `%fpp_user_names%` (comma-joined names of that player's bots). `%fpp_real%` = `Bukkit.getOnlinePlayers().size() - localBots` — bots go through `placeNewPlayer()` so they appear in `getOnlinePlayers()`; FPP subtracts local bot count to get the true real-player count. `%fpp_total%` = `real + bots`.
- LuckPerms: `LuckPermsHelper.subscribeLpEvents(plugin, manager)` is called directly in `onEnable` (not deferred) for auto-prefix/suffix refresh via `UserDataRecalculateEvent`.
- Vercel / Node API: `UpdateChecker` falls back from Modrinth to `https://fake-player-plugin.vercel.app`; `frontend/api/check-update.js` and `frontend/api/status.js` intentionally emit Java-compatible keys such as `remoteVersion`, `version`, `version_number`, and `downloadUrl`.
- Metrics: `util/FppMetrics` wraps FastStats and is conditional via `config.yml` flag (`Config.metricsEnabled()`).
